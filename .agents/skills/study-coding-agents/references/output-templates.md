## 详细学习文档模板

```md
# <主题名称>

## TL;DR
- 用 3-6 条说明这个主题最值得学的点
- 明确这类 agent 解决什么问题
- 明确它和相邻方案的主要差异

## 1. 研究范围
- 研究对象:
- 用户提供资料:
- 本文覆盖维度:
- 未覆盖或证据不足的部分:

## 2. 背景与目标
- 这个 agent / 设计模式出现的背景
- 试图解决的核心问题

## 3. 总体架构
- 高层模块划分
- 关键输入输出
- 可选: 在这里插图并附来源

## 4. 核心运行循环
- 任务接收
- 规划/分解
- 工具调用
- 观察与反馈
- 状态更新
- 终止条件

## 5. 关键机制拆解
### 5.1 Tool Use
### 5.2 Planning
### 5.3 Memory / Context
### 5.4 Prompting / Policies
### 5.5 Safety / Boundaries

## 6. 实现与工程细节
- 代码结构或模块边界
- 与模型、工具、沙箱、文件系统的关系
- 依赖外部服务的部分

## 7. 评测、效果与局限
- 官方或第三方 eval
- 已知失败模式
- 适用边界

## 8. 与其他方案的比较
- 可选。仅在用户要求对比时写

## 9. 可复现的最小子集
- 第一次动手只需要实现什么
- 哪些能力可以后置

## 10. 结论
- 你认为最值得学的 3-5 个点

## Sources
- [标题](链接) - 日期 - 一句话说明价值
```

## 最小 Todo 文档模板

默认用 Markdown checklist。若用户给出更严格的 `todolit` 约定，再覆盖此模板。

```md
# <主题名称> 最小实现 Todo

关联文档: [<主题名称> 学习文档](./<topic-slug>-study.md)

## Phase 1. 补齐输入和边界
- [ ] 确认这次只复现最小闭环，不追求完整功能。见《<主题名称>》的“9. 可复现的最小子集”。
- [ ] 列出必须依赖的外部能力和可以暂缓的能力。见“6. 实现与工程细节”。

## Phase 2. 搭最小架构
- [ ] 定义任务输入、状态对象和主循环骨架。见“3. 总体架构”和“4. 核心运行循环”。
- [ ] 先接入 1 个最关键工具，把 observe -> decide -> act 跑通。见“5.1 Tool Use”。

## Phase 3. 补核心策略
- [ ] 加入最小 planning 或 task decomposition 逻辑。见“5.2 Planning”。
- [ ] 加入最小 context / memory 机制，只保留必要状态。见“5.3 Memory / Context”。

## Phase 4. 验证
- [ ] 用 1-2 个代表性任务验证主循环能闭环。见“7. 评测、效果与局限”。
- [ ] 记录一个失败样例和下一步增强方向。见“10. 结论”。
```

## 交叉引用建议

- 优先使用相对路径链接到学习文档
- 文中提到章节时，直接写中文章节名，便于读者搜索
- 如果 Markdown 渲染环境支持稳定锚点，再补充锚点链接
