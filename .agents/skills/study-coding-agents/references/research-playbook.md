# Research Playbook

## 1. 输入归档

先整理用户明确给出的内容:
- 主题名称
- 用户提供的链接、文件、仓库、论文、截图
- 重点问题或比较维度
- 输出深度与是否需要图片

如果信息不全，先做最保守的假设:
- 默认输出到 `docs/<topic-slug>/`
- 默认写两份文件: `*-study.md` 和 `*-todo.md`
- 默认研究目标是“理解设计 + 提炼最小实现路径”

## 2. 检索优先级

按下面顺序提高信任度:
1. 官方仓库、官方文档、官方博客、论文、维护者演讲
2. 维护者 issue / discussion / release notes / commit history
3. 权威 benchmark、论文复现、严肃技术拆解
4. 社区总结、播客、论坛、社交媒体

使用低优先级来源时:
- 只把它当作发现线索或补充视角
- 核心事实必须回到高优先级来源确认

## 3. 建议搜索维度

除非用户明确收窄范围，否则至少检查这些关键词组合:
- `<topic> architecture`
- `<topic> agent loop`
- `<topic> tool use`
- `<topic> planning`
- `<topic> memory context`
- `<topic> prompt system prompt`
- `<topic> eval benchmark`
- `<topic> repo codebase`
- `<topic> blog talk interview`

如果主题是某个具体产品或仓库，再补:
- `<topic> docs`
- `<topic> README`
- `<topic> issues`
- `<topic> release`
- `<topic> paper`

## 4. 信息抽取框架

为每个高价值来源提取:
- 这是什么来源
- 日期
- 解决什么问题
- 系统的主要环节
- 和其他 agent 的不同点
- 可以复现的最小单元
- 有哪些尚未证实或明显缺失的信息

写作时把信息归到以下桶中:
- 背景与目标
- 总体架构
- 核心运行循环
- 工具和环境交互
- 规划与分解
- 记忆与上下文管理
- 提示词或策略
- 评测与效果
- 实现与工程边界
- 风险、局限与开放问题

## 5. 何时使用 subagent

只有当前用户明确允许 delegation / parallel work，或明确要求广覆盖并行研究时，才启动 subagent。

适合委派的独立子任务:
- 仓库结构与关键模块梳理
- 论文与 talk 的方法提炼
- benchmark / eval / leaderboard 汇总
- 页面截图采集

不要委派:
- 最终判断
- 最终文档写作
- 冲突来源的裁决

委派时保持任务切片互不重叠，并要求返回:
- 核心结论
- 来源链接
- 明确的不确定点

## 6. 何时截图

优先考虑下面这些高信息密度对象:
- 官方架构图
- benchmark 或对比表
- 核心 UI / workflow
- repo 目录图或 README 中的重要流程图

不要截图:
- 纯文字大段落
- 装饰图片
- 与正文无关的页面区域

截图后要补:
- 简短图注
- 原始来源链接
- 为什么这张图值得保留

## 7. 写作标准

- 中文表达，术语前后一致
- 事实和推断明确区分
- 时间敏感事实写日期
- 优先总结和比较，不复制原文
- 引用要能让读者回到原始来源

## 8. 最小 todo 的定义

todo 清单不是“把所有研究点列一遍”。

它应该满足:
- 足够少，能指导第一次最小实现
- 每项都能被完成或验证
- 顺序尽量符合实现依赖
- 每项都能映射回学习文档的某个章节
